# 0008 — RSVP to a meeting

Implements story `docs/user-stories/0008.md`: each upcoming meeting on `/meetings` gets a
name + `wisc.edu`-email RSVP form that POSTs to a new SSR route, validates server-side,
and inserts into the `rsvps` table from 0006. Spec: `SPEC.md` (root, untracked). Built as
pure Astro + one scoped `<script>` — no React, no new dependency, no new migration. This is
the repo's first form→backend pattern; 0009 is expected to reuse it.

## Data flow

```
RsvpForm.astro (scoped <script>)
  → validateRsvp()  (client-side pre-submit feedback)
  → fetch POST /api/rsvp  { name, email, meeting }
      → validateRsvp()  (server is the authority — re-runs the same rules)
      → insertRsvp()  → INSERT into rsvps
          → 23505 unique-violation → duplicate
  ← 201 ok | 400 errors | 409 duplicate | 500 server
```

## Meeting identity

`src/data/meetings.ts` `Meeting` gained a required `id: string` — a URL-safe slug
(convention `YYYY-MM-DD-keyword`, e.g. `2026-09-12-kickoff`). It is the stable identity an
RSVP is keyed to; `rsvps.meeting` stores that slug. The slug also namespaces the form's
element ids (`rsvp-name-<slug>`, …) so multiple forms coexist on one page without id
collisions. Uniqueness is convention only — there is no build-time guard.

## Validation authority split

`src/lib/rsvp-validation.ts` is the single source of the rules: name non-empty after trim,
email matches `/^[^\s@]+@([a-z0-9-]+\.)*wisc\.edu$/i` (local part + optional subdomains +
final `wisc.edu` — so `netid@wisc.edu` and `a@cs.wisc.edu` pass; `foo@gmail.com`,
`foo@wisc.edu.evil.com`, and `@wisc.edu` fail), meeting non-empty. It is pure and
browser-safe (no `node:*`/DB imports), so the client `<script>` imports it for fast inline
feedback and the API route imports the *same* module as the real gate. The client check is
UX only; the server is authoritative.

## API contract — `POST /api/rsvp`

`src/pages/api/rsvp.ts`, `prerender = false`. Every response is a `Response` with an
explicit status and `content-type: application/json`.

| Condition | Status | Body |
|---|---|---|
| Valid, inserted | `201` | `{ ok: true }` |
| Validation errors | `400` | `{ ok: false, errors: [{ field, message }] }` |
| Malformed / absent JSON | `400` | `{ ok: false, code: 'invalid', errors: [] }` |
| Duplicate (email already RSVP'd) | `409` | `{ ok: false, code: 'duplicate' }` |
| Unexpected failure | `500` | `{ ok: false, code: 'server' }` |

Missing/non-string body fields are coerced to empty strings so `validateRsvp` is the single
gate. The 500 path swallows the error detail — the submitted email (PII) is never logged.

## Duplicate handling (23505 → 409)

`src/db/rsvp.ts` `insertRsvp()` inserts and catches, rather than check-then-insert: a
Postgres `unique_violation` (`err.code === '23505'`) on the 0006 `UNIQUE (email, meeting)`
constraint maps to `{ status: 'duplicate' }` → `409`. The DB constraint is the race
arbiter, so concurrent RSVPs can't slip between an existence check and the write. The
`23505` check is isolated in one helper; the exact error shape is assumed against
`@neondatabase/serverless` and confirmed out-of-band against Neon (CI has no live DB — all
DB paths are mocked, per 0006).

## Client enhancement

`src/components/RsvpForm.astro` renders a native `<form method="post" action="/api/rsvp">`
(submittable if JS fails). The scoped `<script>` progressively enhances it:

- **201** → reset the form, announce success in the `role="status" aria-live="polite"` region.
- **400** → render inline field errors (`aria-describedby` + `aria-invalid`), focus the
  first invalid input.
- **409** → open a focus-trapped `role="dialog"` modal (Esc + Close button, focus restored
  to the trigger on close).
- **500 / network** → friendly error, **typed values preserved** for retry.

Note: the no-JS fallback POSTs `application/x-www-form-urlencoded`, which the JSON-only route
rejects with a `400`; full no-JS handling is out of scope for this story.

## Test-override rationale

0007 asserted `/meetings` emits static markup only (`no <form|<input|…>`). 0008 intentionally
supersedes that invariant for upcoming meetings: `tests/meetings.test.ts` now asserts an RSVP
form is present and keyed to each upcoming meeting's slug, and absent from past meetings and
the empty state. This is a deliberate, documented change of the 0007 guarantee — not a
weakened test. Past meetings and the empty state remain form-free.

## Out of scope

No CSRF token or rate-limiting on this public write route, and no public attendee count /
"you're going" state — success is transient. Abuse hardening and richer state are later
concerns.
