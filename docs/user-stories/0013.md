# 0013: Manage calendar events from the admin dashboard

## Story

As an administrator, I want to add, edit, and delete calendar events from the admin dashboard, so that club meetings can be managed without hand-editing code and redeploying.

## Acceptance Criteria

- From `/admin`, an admin can create an event with a date (required), title, time, and location, and on save it persists to Postgres and appears in the admin event list.
- An admin can edit any of an existing event's date, title, time, and location; saved changes persist and replace the prior values in both the admin list and the public page.
- An admin can delete an event, after which it is removed from the admin list and no longer appears on the public `/meetings` page.
- The public `/meetings` page renders events from the database, so any create/edit/delete is reflected on the next visit with no redeploy, preserving the existing upcoming-vs-past split ordering.
- Every event defined today in `src/data/meetings.ts` is migrated into the database so that no historical or upcoming meeting is lost.
- Existing RSVP rows stay correctly associated with their event after the migration (each `rsvps.meeting` value still resolves to the same event).
- Saving an event with a missing date, or a date not in `YYYY-MM-DD` format, shows an inline validation error and does not persist.
- The admin event list shows an empty state when no events exist.
- Unauthenticated / non-admin users cannot reach the event-management routes or mutate events (enforced by the existing 0011 middleware; no new auth logic).

## Technical Notes

- Depends on 0006 (Postgres store, `src/db/schema.ts`, `migrations/`), 0007 (`src/data/meetings.ts`, `src/pages/meetings.astro`, `splitMeetings`), and 0011 (`src/middleware.ts` + `src/lib/admin-guard.ts` gate `/admin/*`). Independent of 0012 (RSVP/inquiry viewing), which proceeds separately.
- **Auth-gate placement is load-bearing, verify first**: `adminRedirect` only gates the `/admin` prefix; `/api/*` is not covered by it (the public `/api/rsvp` route relies on the separate abuse-guard instead). To satisfy "no new auth logic," event mutations must be handled as form POSTs on Astro SSR pages under `/admin/*` (branching on `Astro.request.method`), not as new `/api/*` endpoints — routing them through `/api/*` would require adding auth checks that don't exist today.
- New `events` table via a new file in `migrations/` (e.g. `0003_events.sql`); mirror the shape into `src/db/schema.ts` and extend `tests/db/schema.test.ts`'s sync check.
- New `src/db/event.ts` for list/insert/update/delete, following the thin tagged-template style of `src/db/rsvp.ts` / `src/db/submission.ts`. This is the first per-entity file needing update/delete/list — no existing precedent for those verbs, though the SQL itself is routine.
- New pure validation + deterministic slug generation in a `src/lib/event-validation.ts`-style file (mirroring `rsvp-validation.ts` / `submission-validation.ts`), unit tested with Vitest.
- Slug/RSVP compatibility: the new `events.id` must reuse the same slug scheme currently held in `rsvps.meeting` (no FK exists) so existing RSVP rows keep resolving correctly; no rows exist in `src/data/meetings.ts` today (only commented-out examples), so this is a scheme-compatibility concern rather than a real data-migration risk.
- Convert `src/pages/meetings.astro` from `prerender = true` to per-request SSR (`prerender = false`) so DB changes appear without a redeploy; relocate/repoint the existing pure `splitMeetings()` helper at DB-sourced rows and rework `tests/meetings.test.ts` from mocking `src/data/meetings` to mocking the DB read.
- Delete `src/data/meetings.ts` once its (currently empty) dataset is superseded by the DB — no fallback/seed file kept, per CLAUDE.md's no-redundant-state preference.
- No new dependency, no UI framework (React etc.) — plain `.astro` pages/forms only.
- `pnpm build` (incl. `astro check`) and `pnpm test` must pass, including tests for the new validation/slug logic and the CRUD/list DB helpers.
- Estimated at 8 points — the mandatory-decomposition threshold. A natural two-story split exists (data + public backing vs. admin CRUD UI) if the team wants to de-risk later, but this was deliberately kept as one story by request.
