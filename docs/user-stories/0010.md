# 0010: Protect public submission endpoints from spam and abuse

## Story

As an administrator, I want public submissions filtered against spam and malformed input, so that the RSVP list and inquiry inbox stay trustworthy and usable.

## Acceptance Criteria

- Every public submission (RSVP, join, start-a-digest, inquiry) rejects entries where a hidden honeypot field is filled
- Rapid repeated submissions from the same source are rate-limited and rejected with a clear message
- Submissions with malformed or non-wisc.edu email addresses are rejected server-side, not only client-side
- Rejected submissions are not written to the database
- Legitimate submissions within normal limits continue to succeed

## Technical Notes

- Depends on 0008 and 0009 existing; cross-cutting — must wrap every endpoint consistently
- Honeypot is trivial; rate limiting on serverless requires shared state (KV/Redis/Postgres) — the real cost driver and an infra decision
- Assumption: adding a rate-limit backing store (e.g. Vercel KV / Upstash) is acceptable — revisit if not
- Could alternatively be folded into the acceptance criteria of 0008 and 0009 rather than shipped standalone
- Affected files (create): `src/lib/{honeypot,rate-limit,validate}.ts`, middleware; edit both API routes
